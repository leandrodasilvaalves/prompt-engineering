## Chain-of-Thought Prompting (CoT)

Pré-requisito: `8. One-Shot e Few-Shot Prompting.md`

Chain-of-Thought é uma técnica que instrui o modelo a raciocinar passo a passo antes de chegar à resposta final. Em vez de pedir a conclusão diretamente, você pede o processo de raciocínio.

Isso melhora significativamente a qualidade em tarefas que exigem análise, diagnóstico ou planejamento.

### Quando usar
- Análise de problemas complexos (segurança, troubleshooting, arquitetura)
- Quando a resposta direta tende a ser superficial
- Decisões que precisam de justificativa auditável

### Quando não usar
- Tarefas simples onde o raciocínio intermediário não agrega valor
- Quando tokens são limitados e a resposta direta é suficiente

## Exemplo 1 — Análise de Vulnerabilidade de Segurança

**Cenário:** Nossa aplicação web recebeu um relatório de segurança indicando vulnerabilidade SQL Injection no endpoint `/api/users/search`. O endpoint aceita parâmetro `query` via GET. O código atual usa concatenação de strings para montar a query SQL.

### ❌ Prompt Padrão (Sem CoT)

```markdown
Cenário: Nossa aplicação web recebeu um relatório de segurança indicando
vulnerabilidade SQL Injection no endpoint /api/users/search. O endpoint aceita
parâmetro 'query' via GET. O código atual usa concatenação de strings para
montar a query SQL.

Esta vulnerabilidade é crítica? Recomende ações imediatas.
```

### 🧠 Zero-shot Chain-of-Thought

```
Cenário: Nossa aplicação web recebeu um relatório de segurança indicando
vulnerabilidade SQL Injection no endpoint /api/users/search. O endpoint aceita
parâmetro 'query' via GET. O código atual usa concatenação de strings para
montar a query SQL.

Vamos analisar os riscos passo a passo:
1. Esta vulnerabilidade é crítica?
2. Qual o impacto potencial?
3. Recomende ações imediatas
4. Como prevenir esse tipo de vulnerabilidade no futuro?
```

### 🎯 Few-shot Chain-of-Thought

```
Exemplo de análise de vulnerabilidade estruturada:

Cenário: Buffer overflow em função de login
Análise passo a passo:
1. Severidade: CRÍTICA - permite execução de código arbitrário
2. Impacto: Comprometimento total do sistema
3. Ação imediata: Desabilitar endpoint até correção
4. Prevenção: Code review obrigatório + testes automatizados

Agora analise este novo cenário seguindo a mesma estrutura:

Cenário: SQL Injection no endpoint /api/users/search. O endpoint aceita
parâmetro 'query' via GET. Código usa concatenação de strings.

Análise passo a passo:
```

## Exemplo 2 — Troubleshooting de Kubernetes

**Cenário:** Nosso cluster Kubernetes tem 5 nós worker, mas três pods críticos entraram em estado CrashLoopBackOff após a última atualização. Os logs mostram erro de conexão com PostgreSQL. O banco está em namespace separado e outros serviços acessam normalmente.

### ❌ Prompt sem Chain-of-Thought

```markdown
Nosso cluster Kubernetes tem 5 nós worker, mas três pods críticos entraram em estado
CrashLoopBackOff após a última atualização. Os logs mostram erro de conexão com PostgreSQL.
O banco está em namespace separado e outros serviços acessam normalmente.

O que pode estar causando este problema e como resolver?
```

### 🧠 Zero-shot Chain-of-Thought

```markdown
Nosso cluster Kubernetes tem 5 nós worker, mas três pods críticos entraram em estado
CrashLoopBackOff após a última atualização. Os logs mostram erro de conexão com PostgreSQL.
O banco está em namespace separado e outros serviços acessam normalmente.

Vamos analisar este problema passo a passo.
```

### 🎯 Few-shot Chain-of-Thought

```markdown
Exemplo de troubleshooting Kubernetes estruturado:

Problema: Pods de frontend em CrashLoopBackOff após deploy
Investigação passo a passo:
1. Estado dos pods: `kubectl get pods -o wide` - todos pods em 1 nó específico
2. Logs: `kubectl logs pod-name` - erro de memória insuficiente
3. Recursos: `kubectl describe node node-name` - nó com alta utilização
4. Mudanças: Verificar se resource limits foram alterados no deploy
5. Solução: Ajustar resource requests/limits e redistribuir pods

Problema: Service mesh falhando após upgrade
Investigação passo a passo:
1. Conectividade: Testar comunicação entre pods com `kubectl exec`
2. Configuração: Verificar mudanças nos ConfigMaps e Secrets
3. Versioning: Comparar versões do Istio antes/depois do upgrade
4. Network policies: Checar se políticas de rede bloquearam tráfego
5. Solução: Rollback do service mesh e upgrade incremental

---

Agora analise este novo problema seguindo a mesma estrutura metodológica:

Nosso cluster Kubernetes tem 5 nós worker, mas três pods críticos entraram em estado
CrashLoopBackOff após a última atualização. Os logs mostram erro de conexão com PostgreSQL.
O banco está em namespace separado e outros serviços acessam normalmente.

Vamos diagnosticar este problema sistematicamente passo a passo:

1. Primeiro, analise o estado atual dos pods e identifique padrões no CrashLoopBackOff
2. Em seguida, examine a conectividade de rede entre os namespaces e possíveis mudanças
3. Depois, investigue as configurações que mudaram na última atualização
4. Avalie por que outros serviços conseguem acessar o PostgreSQL normalmente
5. Por fim, proponha uma solução baseada nos achados da investigação

Para cada passo, explique seu raciocínio e os comandos que usaria para investigar.
```

## Exemplo 3 — Refatoração de código

**Cenário:** Analisar e refatorar o `UserService` abaixo que está causando problemas em produção.

```python
import requests
import json
import time

class UserService:
    def __init__(self):
        self.users = []
        self.cache = {}

    def get_user(self, user_id):
        if user_id in self.cache:
            return self.cache[user_id]

        try:
            response = requests.get(f"https://api.users.com/users/{user_id}")
            if response.status_code == 200:
                user_data = json.loads(response.text)
                self.cache[user_id] = user_data
                return user_data
            else:
                return None
        except:
            return None

    def get_users_batch(self, user_ids):
        result = []
        for user_id in user_ids:
            user = self.get_user(user_id)
            if user:
                result.append(user)
            time.sleep(0.1)  # Rate limiting
        return result

    def update_user_status(self, user_id, status):
        user = self.get_user(user_id)
        if user:
            payload = {"status": status}
            response = requests.put(
                f"https://api.users.com/users/{user_id}",
                data=json.dumps(payload),
                headers={"Content-Type": "application/json"}
            )
            if response.status_code == 200:
                if user_id in self.cache:
                    del self.cache[user_id]
                return True
        return False

    def validate_user_data(self, user_data):
        if not user_data.get("email"):
            return False
        if not user_data.get("name"):
            return False
        if len(user_data.get("name", "")) < 2:
            return False
        return True
```

### ❌ Prompt Zero-Shot

```markdown
Contexto: Sou Tech Lead de uma startup de e-commerce. Este código UserService está causando
problemas de performance e instabilidade em produção (timeouts, cache vazando memória,
lentidão com +100 usuários simultâneos).

Preciso de um plano de refatoração para resolver os problemas atuais e preparar para novas
funcionalidades. Tenho 4 semanas para implementar.

Analise este código Python e identifique melhorias necessárias. Crie um plano de refatoração
com prioridades.

```

### 🧠 Prompt Chain-of-Thought

```markdown

Ambiente: Aplicação escalável horizontalmente (3 instâncias), ~50k usuários ativos/dia,
integração crítica com API externa.

Vamos examinar este código sistematicamente:

1. Primeiro, identifique problemas de design e arquitetura que causam os issues reportados
2. Em seguida, analise problemas de performance e memory leaks específicos
3. Depois, examine questões de segurança e tratamento de erros que afetam estabilidade
4. Avalie a observabilidade e capacidade de debugging do código atual
5. Identifique violações de princípios SOLID que dificultam manutenção
6. Por fim, crie um plano de refatoração de 4 semanas com sprints priorizados

Para cada passo, conecte os problemas identificados com os sintomas reportados em produção.
```
