## Self-Consistency

Pré-requisito: `10. Tree of Thoughts - ToT.md`

Self-Consistency é uma técnica onde você executa a mesma análise múltiplas vezes com perspectivas ou focos diferentes, e depois consolida os resultados. O objetivo é aumentar a confiança na conclusão final — se perspectivas independentes convergem para a mesma causa raiz ou decisão, a resposta é mais robusta.

Diferente do Tree of Thoughts (que explora opções em paralelo), o Self-Consistency explora o mesmo problema por ângulos diferentes para validar ou refutar hipóteses.

### Quando usar
- Post-mortems e análise de incidentes complexos com muitos dados
- Decisões de alto impacto onde você quer validar a conclusão por múltiplos ângulos
- Quando uma única perspectiva pode ter viés ou pontos cegos

### Quando não usar
- Problemas simples onde uma análise é suficiente
- Quando tokens e tempo são limitados (Self-Consistency é a técnica mais custosa)

## Exemplo — Post-Mortem Black Friday

> Este exemplo usa o mesmo cenário do arquivo `10. Tree of Thoughts - ToT.md` para demonstrar como diferentes técnicas abordam o mesmo problema. Compare os dois para entender quando usar cada uma.

**Cenário:** Sistema de e-commerce ficou completamente fora do ar por 45 minutos durante a Black Friday (14:15–15:00), causando prejuízo de R$ 500 mil. Sistema atende 15k usuários simultâneos no pico.

**Contexto:** Deploy realizado 16h antes do incidente com otimizações de performance e nova feature de recomendações. Tráfego subiu de 1.200 para 4.100 RPS antes da falha.

```markdown
OBJETIVO: Identificar a causa raiz explorando múltiplas hipóteses de falha.

**Etapa 1 – Exploração de hipóteses:**

Analise, de forma independente:
- Hipótese 1: Infrastructure Overload
- Hipótese 2: Database Bottleneck
- Hipótese 3: Application Memory Leak
- Hipótese 4: External Dependencies Failure

Para cada hipótese:
1. Evidências que suportam esta hipótese (cite logs/métricas específicos)
2. Timeline de falha: como explica a sequência de 14:10 até 14:18?
3. Mecanismo de cascade failure: como levou ao complete outage?
4. Correlação com o deploy: como as mudanças contribuíram?
5. Explicação das anomalias (ex: CDN cache miss spike)
6. Fatores que contradizem esta teoria

**Etapa 2 – Avaliação comparativa:**

Compare as quatro hipóteses:
- Consistência com dados: qual explica melhor todas as evidências?
- Simplicidade: qual requer menos suposições?
- Timeline alignment: qual se alinha melhor com a sequência temporal?
- Correlation strength: qual tem correlação mais forte com o deploy?
- Cascade mechanism: qual explica melhor o complete system failure?

Classifique cada hipótese como Alta/Média/Baixa probabilidade.

**Etapa 3 – Root cause definitiva:**

- Determine a root cause mais provável
- Justifique por que é a mais consistente com os dados
- Identifique fatores contribuintes secundários
- Proponha plano de prevenção e melhorias de monitoramento
```

```markdown
Siga uma abordagem de self-consistency, executando esta análise 3 vezes com focos diferentes:

**EXECUÇÃO 1 - PERSPECTIVA: Infrastructure & Platform**
Foque em métricas de infraestrutura, Kubernetes, recursos computacionais e rede.
Analise: CPU, memória, conexões de rede, limites de recursos, configuração do cluster.

**EXECUÇÃO 2 - PERSPECTIVA: Application & Database**
Foque nos logs de aplicação, queries de database, connection pools e lógica de negócio.

**EXECUÇÃO 3 - PERSPECTIVA: Dependencies & External Services**

1. Causa raiz mais provável baseada nos dados da sua perspectiva
2. Sequência de eventos que levou ao failure completo
3. Evidências específicas que suportam sua hipótese (cite logs/métricas)
4. Fatores contribuintes secundários identificados
5. Recomendações para prevenção baseadas na sua análise

**CONSOLIDAÇÃO FINAL:**
- Compare as 3 análises e identifique pontos de consenso e divergência
- Determine qual perspectiva oferece a explicação mais consistente com todos os dados
- Proponha a causa raiz definitiva e plano de ação para prevenção
```
